Project Synthesis

This section outlines the core objectives and mechanics of "Liminal Sin," a fully featured FMV psychological horror demo built for the Gemini Live Agent Challenge. The visual language now reflects a Las Vegas after-hours signal bleed: neon magenta haze, cyan edge lighting, reflective voids, and casino-floor shadow geometry beneath every panel.

The Experience

  • ⏱
    5-Minute Threshold Loop: Condensed vertical slice where Bally’s realism decays into a flooded under-city of tunnel concrete, broken LEDs, and impossible reflections.
  • ∞
    Stateful Replayability: Trust/Fear and Fourth-Wall counters mutate scene routing, anomaly cadence, and character compliance over repeat runs.
  • 🗣
    Native Barge-In: Players interrupt agents mid-utterance; speech output halts instantly, then resumes with memory of interruption pressure and trust impact.

Implementation Steps (Detailed)

1. Enable APIs + IAM: Enable Vertex AI API, Cloud Run Admin API, Firestore API, and Cloud Build API; create least-privilege service accounts for runtime and deploy phases.
2. Session Gateway: Build a Cloud Run gateway that authenticates client sessions, issues ephemeral tokens, and establishes Gemini Live streaming channels for mic/camera events.
3. GameMaster Router: Route user turns by `proximity_state` and `intent`; write `scene_key`, `player_emotion`, and `slotsky_trigger` into Firestore in a single transactional update.
4. Character Workers: Bind Jason/Audrey/Josh prompts to trust/fear thresholds from Firestore; enforce refusal mechanics when `fear_index` surpasses `trust_level`.
5. Slotsky Dispatcher: Read state flags (`filter: FS`, `fourth_wall_count`, `FOUND`) and enqueue deterministic anomaly events (bells, cards, geometry shifts) with cooldown locks.
6. FMV Storage + Fallback: Serve pre-generated FMV from Cloud Storage; if clip miss occurs, activate fallback loop and let live audio stall while generation catches up.

Cloud System Playbook

Contest-critical setup strategy from clean repo clone to live multimodal orchestration. This is intentionally explicit for reproducible deployments and judge validation.

  1. Project bootstrap: create/select `myceliainteractive` GCP project, attach billing/credits, and set default region closest to target players (`us-west1` or `us-central1`).
  2. Repository baseline: clone repo, install dependencies, run lint/typecheck locally, and validate `.env.local` is gitignored before keys are added.
  3. Secret contract: store `GEMINI_API_KEY`, Firebase config, and Cloudflare tokens in Secret Manager; mount secrets into Cloud Run runtime instead of hardcoding env in source.
  4. Data model: create Firestore `sessions/{sessionId}` documents with sub-objects for `characters`, `trust_level`, `fear_index`, `proximity_state`, `scene_key`, and `slotsky_trigger`.
  5. Live channel wiring: implement WebSocket client transport from browser to Cloud Run gateway; translate VAD segments and barge-in events into Gemini Live request frames.
  6. Agent orchestration: in Cloud Run, process each turn with GameMaster first, then fan out to character agents according to `proximity_state`; persist atomic state write after response generation.
  7. Procedural anomaly flow: trigger Slotsky from deterministic rules (`fourth_wall_count >= 3`, `filter: FS`, trust spikes); publish event IDs consumed by frontend to drive FMV/audio swaps.
  8. Media path: host FMV clips in Cloud Storage, fronted by CDN caching; implement fallback loop when requested `scene_key` clip is absent or late.
  9. Observability: log per-turn timing (`t_vad`, `t_llm`, `t_scene`, `t_net`) to Cloud Logging; chart P95 latency and preserve screenshot proof for contest submission.
  10. Deployment proof: produce final demo with active Cloud Run URL, Firestore state updates visible, and architecture diagram showing client → gateway → Gemini Live → Firestore/Storage loop.
